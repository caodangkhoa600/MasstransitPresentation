# Demo 5: State Machine Saga — Presenter Script

## Hook (~2 min)

> "You have three microservices — Order, Payment, Shipping. A customer clicks Buy. How do you
> coordinate all three without a single giant database transaction?"

Ask the audience: "What happens if Payment succeeds but Shipping is down?" Let them answer.

> "The problem is the distributed transaction problem. You can't lock all three services at once
> the way you lock rows in a single database. Two-phase commit exists, but it couples services at
> the infrastructure level and creates cascading failure risks. Nobody does this in practice."

> "The Saga pattern solves this differently: instead of one transaction, we use a sequence of events.
> Each step publishes an event. The saga tracks which step you're on for each order. If something goes
> wrong, you publish a compensating event — not a rollback, a deliberate undo message."

> "MassTransit's StateMachine Saga is the clean, production-grade way to implement this."

---

## Concept explanation (~3 min)

Draw on a whiteboard or point to the slide:

```
SubmitOrder → [Submitted] → AcceptOrder → [Accepted] → ShipOrder → [Shipped]
                               ↓                         ↓
                          CancelOrder              CancelOrder
                               ↓                         ↓
                           [Cancelled]              [Cancelled]
```

> "Every order gets its own instance of this state machine. Think of it as a row in a table,
> keyed by CorrelationId. When an event arrives with a CorrelationId, MassTransit looks up the
> row for that ID, checks the current state, finds the right `When(...)` clause, runs it, and
> saves the updated state. If no row exists and we're in the `Initially` block, it creates one."

> "The state machine definition is completely separate from the state data. The machine is a
> singleton; there can be thousands of `OrderState` instances — one per active order."

---

## Code walkthrough (~10 min)

### `Contracts/` — the events

Open `SubmitOrder.cs`:
> "Every event contract carries a `CorrelationId`. This is the key that links all events for
> the same order to the same saga instance. It's a `Guid` generated by the publisher."

Show all four contracts briefly. Point out they're pure C# records with zero dependencies.

### `SagaHost/OrderState.cs`

> "This is the row. `CorrelationId` is the key. `CurrentState` is a string that MassTransit
> manages — never set it manually. Everything else is business data we accumulate: the customer
> name when we receive SubmitOrder, the tracking number when we receive ShipOrder."

### `SagaHost/OrderStateMachine.cs`

> "The machine definition. First: `InstanceState(x => x.CurrentState)` — this tells MassTransit
> which property to use as the state field."

> "Then we declare the events and how to correlate them:"
```csharp
Event(() => OrderSubmitted, x => x.CorrelateById(ctx => ctx.Message.CorrelationId));
```
> "CorrelateById means: use this GUID to find the matching OrderState."

Walk through `Initially`, `During(Submitted)`, `During(Accepted)`:
> "`Initially` fires when no instance exists yet — it creates a new one.
> `During(Submitted)` fires only when the current state is Submitted.
> If AcceptOrder arrives when state is Accepted (wrong order), it's simply ignored."

### `SagaHost/Program.cs`

Highlight the key line:
```csharp
x.AddSagaStateMachine<OrderStateMachine, OrderState>()
    .InMemoryRepository();
```
> "One line registers both the state machine definition and where to store instances.
> In production you'd replace `.InMemoryRepository()` with `.EntityFrameworkRepository()`
> or `.RedisRepository()`. The state machine code doesn't change."

> "`ConfigureEndpoints` auto-creates the RabbitMQ queue and binds it to the event type exchanges."

### `Publisher/Program.cs`

> "Two scenarios with separate CorrelationIds — one for the happy path, one for cancellation.
> Every event we publish carries the same CorrelationId within a scenario, so they all map
> to the same saga instance."

## Live demo steps (~10 min)

**Setup:**
1. Start SagaHost terminal — show "Waiting for events"
2. Start Publisher terminal — show the two-scenario introduction

**Scenario A (happy path):**

3. Press Enter → SubmitOrder sent
   - Switch to SagaHost: should see `[SAGA] SubmitOrder → State: Submitted`
4. Press Enter → AcceptOrder sent
   - Switch to SagaHost: should see `[SAGA] AcceptOrder → State: Accepted`
5. Press Enter → ShipOrder sent
   - Switch to SagaHost: should see `[SAGA] ShipOrder → State: Shipped | Tracking: TRACK-ABC-1234`

**Scenario B (cancellation):**

6. Press Enter → SubmitOrder for new order
7. Press Enter → CancelOrder
   - SagaHost: `[SAGA] CancelOrder → State: Cancelled | Reason: Customer changed mind`

**Q&A scenario (memory loss):**

8. Start Scenario A again (restart Publisher or add a third scenario run)
9. After SubmitOrder arrives → **kill SagaHost with Ctrl+C**
10. Restart SagaHost
11. Send AcceptOrder — it arrives but nothing happens: no matching saga instance
> "The state is gone. This is the InMemoryRepository limitation. In production this is why you
> use a persistent repository. And if you also need guaranteed message delivery — where publishing
> an event and saving state are atomic — that's the Outbox pattern. Demo 7."

---

## Key takeaway (30 sec)

> "Saga solves the distributed coordination problem without two-phase commit.
> The state machine tells MassTransit exactly what to do with each event for each order.
> Swap `.InMemoryRepository()` for `.EntityFrameworkRepository()` and your state survives restarts.
> The state machine code itself never changes."
